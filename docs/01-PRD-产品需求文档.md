## 一、产品愿景与定位

### 1.1 产品名称

星途AI (StarPath AI)

Slogan：每个星星都有自己的轨迹

### 1.2 产品定位

一个纯AI驱动的自闭症儿童康复服务平台。用户不再面对复杂的表单、页面和按钮，而是只面对一个懂孩子的AI伙伴和一个动态看板。所有的评估、训练、记录、分析、协作，都通过与AI对话完成。

### 1.3 解决什么问题

自闭症儿童的康复是一个长期、多角色协作的过程。目前家长、医生、特教老师面临的核心痛点：

| 角色 | 痛点 |
|------|------|
| 家长 | 不知道怎么在家训练、记录繁琐容易放弃、看不懂专业评估报告、焦虑无处倾诉、各方信息碎片化 |
| 医生 | 患儿数据不连续（只有就诊时的快照）、无法追踪居家训练效果、与家长/老师沟通成本高 |
| 特教老师 | IEP制定耗时、教学数据记录繁琐、家校信息不同步、缺乏个性化教学建议 |
| 共同痛点 | 三方数据割裂、没有统一的孩子画像、协作靠微信群效率低 |

### 1.4 与传统产品的本质区别

| 维度 | 传统康复管理产品 | 星途AI |
|------|----------------|--------|
| 交互方式 | 表单填写、页面跳转、按钮操作 | 与AI对话，说话就能完成一切 |
| 数据存储 | MySQL/PostgreSQL关系型数据库 | Supabase（PostgreSQL）真源 + child_snapshots 表双层快照（short_term 30天滑动窗口 / long_term 完整发展档案），按child_id组织数据 |
| 智能程度 | AI作为辅助工具（生成报告、回答问题） | AI作为唯一服务入口，Orchestrator自主编排所有能力 |
| 个性化 | 基于规则的推荐 | 基于长期记忆的深度个性化（AI记住每个孩子的偏好、有效策略、触发因素） |
| 权限控制 | 数据库RBAC角色权限表 | care_teams授权 + Supabase RLS行级权限，控制谁能看什么、改什么 |
| 看板 | 后端查数据库渲染固定图表 | AI动态决定展示什么、生成什么洞察 |
| 扩展能力 | 加功能需要改代码、改数据库 | 新增一个 Edge Function 即可扩展新能力，Serverless 弹性伸缩 |

### 1.5 目标市场

| 指标 | 数据 |
|------|------|

| 每年新增确诊 | 约16万 |

## 三、核心设计理念

### 3.1 AI即服务入口

用户打开App后，核心界面只有两个：一个AI对话窗口和一个动态看板。没有复杂的菜单、没有层层嵌套的页面。用户想做任何事，直接跟AI说就行。

传统方式：用户需要找到"评估"菜单 → 选择量表 → 逐题填写 → 提交 → 跳转到结果页
星途方式：用户说"帮我做个M-CHAT筛查" → AI逐题引导对话 → 自动计算 → 直接在对话中给出解读和建议

AI不是一个"功能按钮"，而是一个懂孩子的伙伴。它记得孩子的名字、喜好、有效策略、历史数据，每次对话都是个性化的。

### 3.2 结构化数据库 + 对话语义层

每个孩子在系统中有一个独立的"数据域"（`child_id`），AI通过结构化数据实体获取完整上下文并完成读写。

| 数据实体 | 作用 |
|---------|------|
| children_memory | AI长期记忆（性格、偏好、触发因素、有效策略、里程碑） |
| children_profiles | 六领域能力画像（语言沟通、社交互动、认知学习、感觉运动、情绪行为、生活适应）与版本历史 |
| care_teams | 照护团队与权限配置（家长、医生、老师） |
| assessments | 标准化评估记录（M-CHAT、CARS等） |
| training_plans / training_sessions | 训练方案与训练执行记录 |
| life_records / behavior_records | 生活记录与ABC行为记录 |
| conversations / chat_messages | 对话会话管理与消息持久化（替代Redis会话缓存） |
| reports / notifications | 报告（含周期报告PDF）与跨角色协作通知 |
| Supabase Storage | 图片、语音、视频、周期报告PDF等媒体资源 |
| child_snapshots（short_term / long_term） | 每个孩子一组双层语义快照，用于AI快速加载上下文与跨系统导出 |

采用结构化数据库的核心价值：
- 数据一致性：事务、外键、约束保证数据质量
- 查询效率：SQL + 索引支持多维检索与聚合分析
- 权限可执行：Supabase RLS在数据库层强制权限，不依赖客户端约定
- 可扩展性：新能力通过新增表字段和 Edge Function 扩展，不破坏主链路

同步策略：
- 数据库是权威真源（source of truth）
- `child_snapshots` 表存储双层快照：short_term（30天滑动窗口，高频刷新）和 long_term（完整发展档案含最近3次评估对比，低频刷新）
- 每次关键数据变更后按事件类型异步刷新对应层快照（训练/生活/行为→短期，评估/画像/记忆→长期，权限/危机→两层同时）

### 3.3 Orchestrator + Edge Functions 架构

星途AI采用 Supabase Edge Functions（Deno 运行时）作为全部后端，通过 Orchestrator 入口函数进行意图识别和路由，分发到各业务 Edge Function 执行具体逻辑。

| 层 | 是什么 | 做什么 |
|----|--------|--------|
| Orchestrator | AI的"大脑和调度中心" | 接收用户消息，通过 Doubao-32k-pro 进行意图识别，路由到对应的业务 Edge Function |
| 业务 Edge Functions | AI的"专业能力模块" | 每个 Edge Function 封装一个业务域的完整逻辑（prompt + 模型调用 + 数据库读写） |
| Supabase 数据层 | AI的"记忆和存储" | PostgreSQL 结构化存储 + Storage 媒体托管 + RLS 行级权限 |

Edge Functions 业务模块：

| Edge Function | 职责 | 使用模型 |
|---------------|------|---------|
| orchestrator/ | 入口，意图识别与路由 | Doubao-32k-pro |
| chat-casual/ | 日常对话、情感支持、通用问答 | Doubao-32k-pro |
| assessment/ | 评估筛查（M-CHAT、CARS-2等） | Kimi 2.5 |
| training-record/ | 训练记录结构化提取 | Doubao-32k-pro |
| training-advice/ | 训练建议/方案生成 | Kimi 2.5 |
| behavior-crisis/ | 行为危机识别与应对指导（不降级） | Kimi 2.5 |
| life-record/ | 生活记录（情绪/睡眠/饮食/里程碑） | Doubao-32k-pro |
| dashboard/ | 看板数据分析与洞察生成 | Kimi 2.5 |
| weekly-report/ | 一周训练周报生成 | Kimi 2.5 |
| analysis-report/ | 综合分析报告生成 | Kimi 2.5 |
| collaboration/ | 协作通知、跨角色消息 | Kimi 2.5 |
| periodic-report/ | 周期报告（3/6/12个月），生成PDF存Supabase Storage | Kimi 2.5 |
| snapshot-refresh/ | 快照刷新Worker（无模型，纯数据聚合） | 无 |

工作流程：用户消息 → Orchestrator 意图识别 → 路由到对应业务 Edge Function → Edge Function 加载角色 prompt + 调用模型 → 结果写入 Supabase → 触发 snapshot-refresh → Streamable HTTP 流式回复用户

写入完成约束（Final Writeback）：
- 所有写入类动作完成后，必须写入操作审计日志并写入快照刷新事件。
- 主业务写入与刷新事件写入必须同事务提交，避免"写入成功但快照未刷新"。

### 3.4 多模态输入

用户可以通过文字、图片、语音、视频与AI交互：

| 输入类型 | 处理方式 | 典型场景 |
|---------|---------|---------|
| 文字 | 直接进入 Orchestrator 处理 | 日常对话、提问、记录 |
| 图片 | 多模态模型（Kimi 2.5/Doubao 2.0）分析后转为文字描述+原图一起给业务 Edge Function | 发送训练场景照片、行为记录截图、评估量表照片 |
| 语音 | ASR语音识别转文字后给 Orchestrator | 训练过程中语音记录、开车时语音提问 |
| 视频 | 抽取关键帧+语音转文字后给业务 Edge Function | 发送孩子训练视频让AI分析行为表现 |

### 3.5 不同角色不同Agent

三种角色面对的是完全不同"人格"的AI：

| 维度 | 家长Agent | 医生Agent | 特教老师Agent |
|------|----------|----------|-------------|
| 称呼 | "您"和孩子昵称 | 简洁直接 | "老师您好" |
| 语气 | 温暖共情，先回应情绪再给建议 | 专业严谨，数据驱动 | 实操导向，步骤清晰 |
| 术语 | 通俗易懂，避免专业术语 | 使用医学术语，附必要解释 | 使用教育术语，配实操说明 |
| 回复结构 | 共情→分析→建议→鼓励 | 数据→判断→建议→证据等级 | 目标→步骤→材料→注意事项 |
| 禁忌 | 不说"正常孩子"，用"同龄典型发展儿童" | 不做确诊性结论，用"提示""建议" | 不否定家长努力 |
| 特有能力 | 情绪支持、居家指导、危机应对 | 诊断辅助、鉴别诊断、用药参考 | IEP制定、课堂策略、行为管理 |

## 九、风险与应对

| 风险 | 影响 | 概率 | 应对策略 |
|------|------|------|---------|
| AI回复不准确/不专业 | 用户信任度下降，可能造成误导 | 中 | 各业务 Edge Function 内置专业 prompt 约束输出、专业人士审核 prompt 内容、明确AI安全边界（不做诊断） |
| AI响应速度慢 | 用户体验差 | 中 | 双模型策略（Doubao-32k-pro 快速响应 + Kimi 2.5 专业场景）、Streamable HTTP 流式输出、Orchestrator 快速路由 |
| 数据库热点与慢查询 | 用户量大时查询变慢、报表延迟 | 中 | 建索引、冷热分层、异步报表任务（periodic-report Edge Function）、慢查询审计 |
| 快照与数据库漂移 | AI读到过期语义，解释偏差 | 中 | 双层快照独立刷新 + is_stale 标记 + 定时全量对账 + 差异告警与自动回补 |
| 儿童数据泄露 | 法律风险、用户信任崩塌 | 低 | 加密存储、Supabase RLS 严格权限控制、安全审计、合规认证 |
| 用户付费意愿低 | 商业模式不成立 | 中 | 免费版提供足够价值建立信任、付费功能差异化明显、机构B2B补充收入 |
| App Store审核不通过 | 上线延迟 | 中 | 提前了解审核规则、医疗类App需要的资质提前准备、小程序作为备选渠道 |
| 多模态模型成本高 | 运营成本压力 | 中 | 按VIP等级控制多模态使用、图片/视频压缩后处理、Doubao-32k-pro 承担高频低成本场景 |
| 家长使用门槛 | 部分家长不习惯AI对话 | 中 | 快捷操作栏降低门槛、新手引导流程、语音输入降低打字负担 |
| Edge Function 冷启动延迟 | 首次请求响应慢 | 低 | 关键函数保持预热、Orchestrator 轻量化减少冷启动影响、Streamable HTTP 流式缓解感知延迟 |
| Supabase 平台依赖 | 平台故障影响全部服务 | 低 | 数据定期导出备份、核心业务逻辑可迁移到其他 Deno/Node 运行时、监控 Supabase 状态页 |
