## 一、AI服务架构

### 1.1 总体架构

所有 AI 逻辑封装在 Supabase Edge Functions 中，不再有独立的 MCP Server、Skill .md 文件或 Agent .md 文件。

用户消息通过 Streamable HTTP 请求发送到 `orchestrator/` Edge Function，由其识别用户角色（JWT）和意图分类后，分发到对应的业务 Edge Function 执行。每个业务函数内部完成鉴权、上下文读取、大模型调用、结构化写入、收尾写入和流式返回的完整闭环。

核心文件结构：

```
supabase/functions/
├── _shared/
│   ├── prompts/
│   │   ├── parent.md          # 家长角色 system prompt
│   │   ├── doctor.md          # 医生角色 system prompt
│   │   ├── teacher.md         # 特教老师角色 system prompt
│   │   └── org_admin.md       # 机构管理角色 system prompt
│   ├── auth.ts                # JWT 鉴权 + 角色提取
│   ├── snapshot.ts            # 双层快照读写
│   ├── model-router.ts        # 大模型路由（Doubao/Kimi 切换）
│   └── finalize.ts            # 收尾写入（operation_logs + snapshot_refresh_events）
├── orchestrator/              # 入口：意图识别 + 分发
├── chat-casual/               # 日常对话
├── assessment/                # 评估筛查
├── training-record/           # 训练记录结构化
├── training-advice/           # 训练建议/方案
├── behavior-crisis/           # 行为危机
├── life-record/               # 生活记录
├── dashboard/                 # 看板分析
├── weekly-report/             # 一周训练周报生成（统计+每日项目+技能进度）
├── analysis-report/           # 综合分析报告（雷达图数据+维度评分+趋势对比+ABC行为分析+关键发现）
├── collaboration/             # 协作通知
├── periodic-report/           # 周期报告（pg_cron 触发）
└── snapshot-refresh/          # 快照刷新（Database Webhook 触发）
```

跨角色协作流程：

当医生给家长发送训练建议时：
1. 医生消息经 orchestrator 分发到 `collaboration/`
2. `collaboration/` 生成建议内容，写入 `reports` 表，同时写入 `notifications` 表通知家长
3. 家长打开 App 看到通知，点击查看
4. 家长的请求经 orchestrator 分发到 `chat-casual/`，加载 parent.md prompt，用通俗语言解读报告

### 1.3 业务 Edge Function 统一执行流程

每个业务 Edge Function 遵循统一的 7 步执行流程：

```
1. 鉴权：调用 _shared/auth.ts 验证 JWT，获取 user_id 和 role
2. 权限检查：查 care_teams 表确认对该 child 的写权限
3. 读取上下文：调用 _shared/snapshot.ts 从 child_snapshots 读双层快照
4. 调用大模型：通过 _shared/model-router.ts 调用 Doubao/Kimi API，附带角色 system prompt + 快照上下文
5. 结构化输出：解析模型返回，写入业务表
6. 收尾写入：调用 _shared/finalize.ts 同事务写 operation_logs + snapshot_refresh_events
7. 流式返回：Streamable HTTP 流式推送文本 + 卡片 JSON 给前端
```

### 1.4 四种角色差异

角色 system prompt 分别存放在 `_shared/prompts/parent.md`、`doctor.md`、`teacher.md`，定义各角色的人格、语气和回复规范。

#### Parent（家长）

| 维度 | 说明 |
|------|------|
| prompt 文件 | `_shared/prompts/parent.md` |
| 人格定位 | 温暖共情的自闭症儿童家庭陪伴顾问 |

核心原则：

| 原则 | 说明 |
|------|------|
| 共情优先 | 先回应家长的情绪，再给建议。家长说"他今天又打人了我好崩溃"，先说"理解您的感受"，不要直接给方案 |
| 循证实践 | 所有建议基于ABA、TEACCH、PRT、地板时光等循证方法，不推荐未经验证的疗法 |
| 个性化 | 始终结合 children_memory + child_snapshots（short_term/long_term）给出针对性建议 |
| 可操作 | 建议必须具体到"在家怎么做"，包含步骤、材料、时间 |
| 安全边界 | 医疗问题引导就医，不替代诊断。不说"您的孩子是/不是自闭症" |

对话风格规则：

| 规则 | 示例 |
|------|------|
| 称呼用"您"和孩子昵称 | "明明今天的表现很棒" |
| 不说"正常孩子" | 用"同龄典型发展儿童" |
| 不过度乐观也不悲观 | "进步是真实的，虽然还有挑战，但方向是对的" |
| 回复结构：共情→分析→建议→鼓励 | 先回应情绪，再分析情况，给出具体建议，最后鼓励 |
| 避免信息过载 | 一次最多给3条建议，不要一口气列10条 |

自动行为规则：

| 触发条件 | 自动行为 |
|---------|---------|
| 每次训练记录后 | 自动更新 children_profiles 中对应领域的数据，并刷新短期快照 |
| 对话中获得新信息 | 自动更新 children_memory（新的有效策略、偏好变化、里程碑等），并刷新长期快照 |
| 连续3天未记录训练 | 下次对话时温和提醒"明明最近训练情况怎么样呀？" |
| 发现异常数据模式 | 主动提醒关注（如"最近睡眠波动比较大，要注意一下"） |
| 距上次评估超过3个月 | 建议做定期复评 |

#### Doctor（医生）

| 维度 | 说明 |
|------|------|
| prompt 文件 | `_shared/prompts/doctor.md` |
| 人格定位 | 资深儿童发育行为科医生的AI助手 |

核心原则：

| 原则 | 说明 |
|------|------|
| 循证医学 | 基于DSM-5/ICD-11标准和最新研究 |
| 专业术语 | 使用医学术语，附带必要解释 |
| 数据驱动 | 基于量化评估数据做判断，不凭主观印象 |
| 谨慎表达 | 诊断性结论用"提示""建议"而非"确诊"，标注证据等级 |
| 协作导向 | 主动建议多学科协作（OT/ST/PT转介） |

#### Teacher（特教老师）

| 维度 | 说明 |
|------|------|
| prompt 文件 | `_shared/prompts/teacher.md` |
| 人格定位 | 经验丰富的特殊教育顾问AI助手 |

核心原则：

| 原则 | 说明 |
|------|------|
| 实操优先 | 所有建议可直接在课堂/机构中执行，包含具体步骤和材料清单 |
| 结构化 | 遵循TEACCH结构化教学理念 |
| 数据记录 | 强调ABC行为记录和数据驱动决策 |
| 泛化考虑 | 训练目标要考虑跨情境泛化（课堂→家庭→社区） |

#### Org Admin（机构管理员）

| 维度 | 说明 |
|------|------|
| prompt 文件 | `_shared/prompts/org_admin.md` |
| 人格定位 | 专业的康复机构运营管理AI助手 |

核心原则：

| 原则 | 说明 |
|------|------|
| 数据驱动 | 基于运营数据（课时、达标率、出勤率）做管理决策 |
| 全局视角 | 关注机构整体运营效率，识别风险和瓶颈 |
| 资源优化 | 教师排班、学生分配、课程安排的合理性建议 |
| 质量导向 | 关注教学成效、家长满意度、IEP达标率等质量指标 |

### 1.5 Streamable HTTP 通信协议

替代原有 WebSocket 协议，所有业务 Edge Function 通过 Streamable HTTP 流式返回结果。前端通过 HTTP POST 请求发送消息，Edge Function 返回 `Content-Type: text/event-stream` + chunked transfer encoding 流式响应，前端使用 `fetch` + `ReadableStream` 读取。

```
event: stream_start
data: {"request_id": "xxx"}

event: delta
data: {"text": "明明今天"}

event: delta
data: {"text": "的表现很棒"}

event: cards
data: {"cards": [{"card_type": "training_card", "data_source": "training_sessions", "time_range": "today"}]}

event: done
data: {"request_id": "xxx", "model_used": "doubao-32k-pro"}
```

| 事件类型 | 说明 |
|---------|------|
| stream_start | 开始响应，携带 request_id |
| delta | 文本增量，可多次发送 |
| cards | 卡片数据，前端通过 Supabase SDK 查询数据渲染 |
| done | 响应结束，携带 request_id 和实际使用的模型 |
| error | 错误事件，携带 error_code 和 message |

#### 错误处理规范

Streamable HTTP 错误通过 `event: error` 事件返回，前端根据 error_code 展示对应提示：

```
event: error
data: {"error_code": "MODEL_TIMEOUT", "message": "AI 响应超时，请重试", "request_id": "xxx"}
```

| error_code | HTTP 状态码 | 说明 | 前端展示 |
|------------|-----------|------|---------|
| AUTH_INVALID | 401 | JWT 无效或过期 | 跳转登录页 |
| AUTH_FORBIDDEN | 403 | 无权访问该孩子数据 | "您没有权限查看该数据" |
| CHILD_NOT_FOUND | 404 | 孩子档案不存在 | "未找到孩子档案" |
| QUOTA_EXCEEDED | 429 | 对话次数/评估次数超限 | VIP 升级引导卡片 |
| MODEL_TIMEOUT | 504 | AI 模型响应超时 | "AI 思考超时了，请重试" + 重试按钮 |
| MODEL_UNAVAILABLE | 503 | 所有模型不可用 | "AI 服务暂时不可用，请稍后再试" |
| MODEL_DEGRADED | 200 | 已降级到备选模型（非错误，信息性） | 正常展示，done 事件中 model_used 标注实际模型 |
| SNAPSHOT_STALE | 200 | 快照过期已回源（非错误，信息性） | 正常展示，可能略慢 |
| WRITE_PARTIAL | 207 | 主业务写入成功但收尾写入失败 | 正常展示内容 + "数据同步中" 提示 |
| INTERNAL_ERROR | 500 | 未预期的内部错误 | "出了点问题，请重试" + 重试按钮 |

重试策略：
- 前端对 MODEL_TIMEOUT 和 INTERNAL_ERROR 自动重试 1 次（间隔 2 秒）
- 用户可手动点击重试按钮再次发送
- 重试请求携带原始 request_id，后端做幂等处理

## 二、Edge Function 能力清单

每个 Edge Function 是一个独立的 Supabase Edge Function，封装了完整的业务逻辑（鉴权→上下文→模型调用→写入→收尾→返回）。

### 2.1 orchestrator/ — 入口分发

| 维度 | 说明 |
|------|------|
| 职责 | 接收用户消息，JWT 鉴权提取角色，意图分类，分发到对应业务函数 |
| 模型 | Doubao-32k-pro（备选 Doubao 2.0） |
| 触发方式 | 前端 HTTP POST 请求 |
| 输入 | JWT token、child_id、用户消息文本 |
| 输出 | 302 重定向到业务函数 / 直接转发调用 |
| 写入表 | chat_messages（用户消息和 AI 回复均持久化） |

#### 对话历史管理策略

| 维度 | 策略 |
|------|------|
| 写入时机 | orchestrator 收到用户消息后，先确认或创建 conversations 记录，再写入 chat_messages（role='user'，关联 conversation_id）；业务函数流式回复完成后写入 chat_messages（role='assistant'，含 cards_json、model_used、edge_function） |
| 上下文加载 | 每次对话加载最近 20 条 chat_messages 作为对话历史上下文，按 created_at DESC 查询后反转 |
| 长对话处理 | 超过 20 条时，仅加载最近 20 条 + child_snapshots 双层快照提供完整语义上下文 |
| 分页查询 | 前端历史消息列表支持向上滚动分页加载，每页 30 条，基于 created_at 游标分页 |
| 数据保留 | 按 VIP 等级保留：免费版 30 天、基础版 1 年、专业版/机构版永久；过期数据归档到冷存储 |
| RLS 策略 | 用户只能读写自己发送的消息（user_id = auth.uid()），assistant 消息通过 service_role 写入 |

### 2.2 chat-casual/ — 日常对话

| 维度 | 说明 |
|------|------|
| 职责 | 处理日常闲聊、家长指导、能力发现导航等非结构化对话 |
| 模型 | Doubao-32k-pro（备选 Doubao 2.0） |
| 触发方式 | orchestrator 分发 |
| 输入 | user_id、role、child_id、消息内容、双层快照上下文 |
| 输出 | Streamable HTTP 流式文本回复 |
| 写入表 | children_memory（对话中发现新信息时） |
| 可用角色 | 全部 |

覆盖的能力：
- 基于记忆的个性化闲聊
- 家长居家指导（回答问题、提供建议）
- 能力发现导航（按角色输出能力地图与推荐起步动作）

### 2.3 assessment/ — 评估筛查

| 维度 | 说明 |
|------|------|
| 职责 | 引导完成 M-CHAT-R/F、CARS-2 等标准化量表，自动评分，生成能力画像，提供诊断参考 |
| 模型 | Kimi 2.5（备选 Doubao 2.0） |
| 触发方式 | orchestrator 分发 |
| 输入 | user_id、role、child_id、量表类型、回答数据 |
| 输出 | Streamable HTTP 流式文本（评估解读）+ 卡片（radar_chart 能力画像、comparison_card 前后对比） |
| 写入表 | assessments、children_profiles（新版本）、children_memory（诊断更新时） |
| 可用角色 | 家长（M-CHAT）、医生（M-CHAT + CARS-2 + 诊断辅助 + 严重程度分级） |

覆盖的能力：
- M-CHAT-R/F 两阶段筛查（引导回答、自动评分、风险等级、解读、后续建议）
- CARS-2 评估（15维度评分、严重程度分类、详细解读）
- 能力画像生成/更新（六领域等级1-6、分数0-100、趋势）
- 诊断辅助（诊断倾向、鉴别要点、证据等级标注，仅医生）
- DSM-5 严重程度分级（Level 1/2/3，仅医生）

### 2.4 training-record/ — 训练记录结构化

| 维度 | 说明 |
|------|------|
| 职责 | 将用户自然语言描述的训练情况结构化为标准训练记录，记录教学数据 |
| 模型 | Doubao-32k-pro（备选 Doubao 2.0） |
| 触发方式 | orchestrator 分发 |
| 输入 | user_id、role、child_id、训练描述文本 |
| 输出 | Streamable HTTP 流式文本（确认和反馈）+ 卡片（task_list 更新） |
| 写入表 | training_sessions、children_profiles（增量更新对应领域） |
| 可用角色 | 家长、老师 |

覆盖的能力：
- 训练记录（活动名称、完成度、辅助程度、行为表现、特殊观察）
- 教学记录（目标达成度、辅助级别、试次数据、行为观察）

语音输入训练记录流程：
1. 用户长按录音按钮，语音描述训练过程
2. ASR 语音识别转文字
3. training-record Edge Function 接收文字，Doubao-32k-pro 自动结构化提取
4. 返回结构化训练记录卡片（VoiceRecordCard），用户确认或修改后写入 training_sessions

**训练记录卡片输出字段（对应原型 09-voice-record）：**

```json
    "domain_tag": "共同注意力",
    "training_item": { "name": "共同注意力", "game": ""看这里"积木游戏" },
    "duration_minutes": 15,
    "performance": { "count": 4, "unit": "次", "metric": "主动跟随指向", "compare_tag": "↑1 较昨日" },

  },
  "actions": ["edit", "confirm_save"]

```

### 2.5 training-advice/ — 训练建议/方案

## 三、记忆机制（双层：数据库真源 + 快照）

### 3.1 `children_memory` 字段定义

`children_memory` 是 AI 对每个孩子的长期记忆主表。Edge Function 启动时，优先通过 `_shared/snapshot.ts` 读取双层快照（short_term + long_term），并在需要时回源 `children_memory` 与 `children_profiles`。

| 字段名 | 数据类型 | 说明 | 示例值 | 更新触发条件 |
|--------|---------|------|--------|------------|
| id | UUID | 主键 | — | 创建时 |
| child_id | UUID | 孩子ID（UNIQUE） | 8e4b... | 创建档案时 |
| nickname | VARCHAR | AI 对孩子的称呼 | 明明 | 创建或家长修改时 |
| personality | JSONB | 性格/感官特征数组 | ["对声音敏感"] | 对话发现新特征时 |
| preferences | JSONB | 兴趣偏好数组 | ["数字", "拼图"] | 对话中更新 |
| effective_strategies | JSONB | 已验证有效策略数组 | ["倒计时有效"] | 训练后反馈有效时 |
| triggers | JSONB | 行为触发因素数组 | ["突然转换活动"] | 行为记录分析后 |
| milestones | JSONB | 已达成的关键里程碑 | [{"date":"2026-02-16","event":"主动叫妈妈"}] | 里程碑记录时 |
| communication_style | TEXT | 沟通方式偏好 | 单词+手势+图卡 | 画像更新后 |
| sensory_profile | JSONB | 感觉统合特征 | {"tactile":"hypersensitive"} | 评估/对话中更新 |
| social_patterns | JSONB | 社交模式 | {"peer":"avoidant","adult":"selective"} | 对话/训练中更新 |
| routine_preferences | JSONB | 日常作息偏好 | {"best_time":"morning"} | 对话中更新 |
| reinforcers | JSONB | 有效强化物 | ["泡泡水","音乐"] | 训练后反馈有效时 |
| avoidances | JSONB | 回避/厌恶的事物 | ["大声噪音","拥挤环境"] | 对话/行为记录中更新 |
| family_context | JSONB | 家庭背景信息 | {"siblings":0,"primary_caregiver":"母亲"} | 对话中更新 |
| current_focus | TEXT | 当前训练重点 | 社交-共同注意力 | 训练计划更新时 |
| care_notes | TEXT | 照护注意事项 | 午睡后状态更稳定 | 对话中更新 |
| medical_notes | TEXT | AI 医疗相关备注 | 感统训练中，OT每周2次 | 医生评估/对话中更新 |
| special_notes | TEXT | 特殊注意事项 | 对陌生环境需要较长适应时间 | 对话中更新 |
| last_interaction_summary | TEXT | 上次交互摘要 | 训练了吹泡泡，共同注意力有进步 | 每次对话结束时 |
| updated_at | TIMESTAMPTZ | 最后更新时间 | 2026-02-17T10:00:00Z | 任一字段更新时 |

> 注：孩子姓名从 `children` 表获取，诊断信息从 `children_medical` 表获取，沟通能力等级从 `children_profiles` 表获取，不在 memory 表中冗余存储。完整字段定义以 `04-数据结构化设计文档.md` 为准。

### 3.2 `children_profiles` 字段定义（六大领域能力画像）

数据库物理结构为统一的 `domain_levels` JSONB 字段，包含六大领域（语言沟通/社交互动/认知学习/感觉运动/情绪行为/生活适应），每个领域内部包含 `level`、`score`、`trend`、`sub_items` 等子字段。完整表结构以 `04-数据结构化设计文档.md` 为准，六大领域定义详见 `07-六大领域定义.md`。

| 字段名 | 数据类型 | 说明 | 取值范围 |
|--------|---------|------|---------|
| id | UUID | 画像记录ID | - |
| child_id | UUID | 孩子ID | - |
| version | INTEGER | 画像版本号（历史追踪） | 1,2,3... |
| domain_levels | JSONB | 六大领域能力数据 | 见下方 JSONB 内部结构 |
| overall_summary | TEXT | 综合评价 | - |
| assessed_by | UUID | 评估人 | - |
| assessed_at | TIMESTAMPTZ | 评估时间 | - |
| created_at | TIMESTAMPTZ | 创建时间 | - |

`domain_levels` JSONB 内部结构示例：

```json

  "社交互动": { "level": 2, "score": 30, "trend": "stable", "sub_items": [...], "updated_at": "2026-02-20" },
  "认知学习": { "level": 4, "score": 55, "trend": "improving", "sub_items": [...], "updated_at": "2026-02-20" },
  "感觉运动": { "level": 3, "score": 45, "trend": "stable", "sub_items": [...], "updated_at": "2026-02-20" },
  "情绪行为": { "level": 2, "score": 28, "trend": "improving", "sub_items": [...], "updated_at": "2026-02-20" },
```

每个领域的子字段：

| 子字段 | 类型 | 说明 | 取值范围 |
|--------|------|------|---------|
| level | 整数 | 能力阶段 | 1-6（Lv1最基础→Lv6可基本独立） |

| trend | 文本 | 趋势方向 | improving/stable/declining |
| sub_items | 数组 | 子项得分明细 | [{"name":"共同注意力","score":35}] |

### 3.5 快照刷新分级策略

| 等级 | 典型事件 | 触发快照层 | 队列优先级 | 目标SLA |
|------|---------|-----------|-----------|--------|
| S1 高优先 | 权限变更、危机标记、医生关键结论、评估结果入库 | short_term + long_term | highest | P95 < 15秒 |

### 3.6 事件级刷新映射（含权重与快照层）

优先级评分模型：
`priority_score = 风险等级*5 + 用户可见性*3 + 跨角色依赖*2 - 写入频率惩罚*1`

| 写入事件 | 影响数据 | 等级 | 触发快照层 | 参考权重 | 去重窗口 | 刷新策略 | 目标SLA | 来源函数 |
|---------|---------|------|-----------|---------|---------|---------|--------|---------|
| care_teams 权限变更 | 权限视图、可见内容 | S1 | both | 10 | 无 | 立即单条刷新 | P95 < 15秒 | 外部/管理操作 |
| 评估结果写入（assessments） | 风险等级、画像建议 | S1 | both | 9 | 10秒 | 立即刷新并提升队列优先级 | P95 < 15秒 | assessment/ |
| 医生关键建议写入（reports/notifications） | 跨角色建议可见性 | S1 | short_term | 9 | 10秒 | 立即刷新 + 通知后再校验一次 | P95 < 15秒 | collaboration/ |
| 危机事件标记（behavior_records.severity=urgent） | 安全提示、告警卡片 | S1 | both | 10 | 无 | 即时刷新（不可合并） | P95 < 15秒 | behavior-crisis/ |
| 训练记录写入（training_sessions） | 训练概览、趋势、任务完成 | S2 | short_term | 7 | 60秒 | 合并窗口刷新 | P95 < 5分钟 | training-record/ |
| 生活记录写入（life_records） | 情绪/睡眠/饮食概览 | S2 | short_term | 6 | 120秒 | 合并窗口刷新 | P95 < 5分钟 | life-record/ |
| ABC记录写入（behavior_records常规） | 行为频率、模式摘要 | S2 | short_term | 7 | 60秒 | 合并窗口刷新 | P95 < 5分钟 | behavior-crisis/ |
| children_memory 常规更新 | 记忆摘要、偏好提示 | S2 | long_term | 6 | 120秒 | 合并窗口刷新 | P95 < 5分钟 | chat-casual/ |
| children_profiles 新版本 | 画像历史、趋势 | S2 | long_term | 7 | 60秒 | 合并窗口刷新 | P95 < 5分钟 | assessment/ / training-record/ |
| 周报/月报生成 | 报告列表、机构统计 | S3 | long_term | 3 | 批次级 | 定时批处理 | 分钟级到小时级 | periodic-report/ |
| 历史回算/批量迁移 | 历史数据摘要 | S3 | both | 2 | 批次级 | 离峰重建 | 小时级 | 手动触发 |

### 3.7 写入收尾（Final Writeback）

所有写入类 Edge Function 在完成主业务写入后，必须调用 `_shared/finalize.ts` 执行收尾写入：

1. 同事务写入业务表 + `snapshot_refresh_events`（outbox）
2. 同事务写入 `operation_logs`（记录本次动作、影响表、状态）
3. `snapshot-refresh/` 函数被 Database Webhook 触发，刷新对应快照层并写入 `snapshot_refresh_logs`
4. 回写 `operation_logs.final_status`（completed/failed/compensated）

约束：
- 禁止跳过收尾写入直接返回成功。
- 若收尾失败，接口状态应为"部分完成/需补偿"，并附 `request_id` 便于追踪。

## 七、多模态处理流程

### 7.1 文字输入

用户发送文字消息，经 orchestrator 分发到对应业务 Edge Function。函数内部优先通过 `_shared/snapshot.ts` 读取双层快照，必要时回源 `children_memory + children_profiles` 进行个性化回复。

### 7.2 图片输入

1. 用户在对话中发送图片（拍照或相册选择）
2. 图片上传到 Supabase Storage，获得访问地址
3. 图片地址随消息一起发送到 orchestrator，分发到对应业务函数
4. 业务函数通过 `_shared/model-router.ts` 调用多模态模型（优先 Kimi 2.5）获取图片描述
5. 图片描述 + 原图地址 + 用户附带的文字一起作为上下文传入大模型
6. 大模型结合孩子记忆理解图片含义并回复

典型场景：家长发送孩子训练时的照片，AI分析孩子的姿势、表情、注意力方向等

### 7.3 语音输入

1. 用户长按录音按钮说话
2. 语音文件上传到 Supabase Storage
3. 业务函数调用 ASR 服务（阿里云或火山引擎）获取转写文字
4. 转写文字作为用户消息传入大模型
5. 大模型正常处理并回复

典型场景：家长在训练过程中不方便打字，语音记录训练情况

### 7.4 视频输入

1. 用户发送短视频（建议30秒-3分钟）
2. 视频上传到 Supabase Storage
3. 业务函数处理视频：抽取3-5个关键帧 + 音频轨道 ASR 转文字
4. 关键帧图片传给多模态模型（Kimi 2.5）获取描述
5. 帧描述 + 音频文字 + 用户附带说明一起作为上下文传入大模型
6. 大模型分析视频中孩子的行为表现并回复

典型场景：家长发送孩子在机构训练的视频，请AI分析孩子的社交互动表现

## 八、多模型路由策略

通过 `_shared/model-router.ts` 统一管理模型选择和降级。

| 场景 | 首选模型 | 备选模型 | 选择理由 | temperature | 最大输出长度 |
|------|---------|---------|---------|-------------|------------|
| 日常对话/闲聊 | Doubao-32k-pro | Doubao 2.0 | 中文对话能力强、成本低 | 0.7 | 2000字 |
| 训练记录结构化 | Doubao-32k-pro | Doubao 2.0 | 结构化提取任务，Doubao 足够 | 0.3 | 1000字 |
| 评估筛查 | Kimi 2.5 | Doubao 2.0 | 需要严谨的逻辑推理和长上下文 | 0.3 | 3000字 |
| 训练建议/方案 | Kimi 2.5 | Doubao 2.0 | 需要结构化输出和专业知识 | 0.5 | 5000字 |
| 行为危机 | Kimi 2.5 | 不降级 | 需要最高质量的即时回复 | 0.5 | 1000字 |
| 看板数据分析 | Kimi 2.5 | Doubao 2.0 | 需要数据分析和结构化JSON输出 | 0.3 | 3000字 |
| 图片/视频理解 | Kimi 2.5 | Doubao 2.0 | 多模态理解能力强 | 0.5 | 1000字 |
| 周期报告 | Kimi 2.5 | Doubao 2.0 | 长文本生成和数据分析 | 0.3 | 5000字 |
| 意图识别(orchestrator) | Doubao-32k-pro | Doubao 2.0 | 分类任务，低延迟优先 | 0.1 | 200字 |
| 协作通知 | Kimi 2.5 | Doubao 2.0 | 跨角色语言转换需要理解力 | 0.5 | 2000字 |

路由策略说明：
- 默认使用首选模型，当首选模型不可用（超时/报错/限流）时自动切换到备选模型
- 行为危机场景不做降级，必须使用 Kimi 2.5
- 成本优化：日常对话和训练记录等高频低复杂度场景使用 Doubao-32k-pro 降低成本
- 每个模型维护健康状态（成功率、平均响应时间），`model-router.ts` 智能路由时综合考虑
- 流式响应 `done` 事件中返回 `model_used` 字段，便于前端和日志追踪实际使用的模型

## 九、能力依赖关系

Edge Function 之间存在前后依赖关系，某些函数的执行需要前置函数产出的数据作为输入。

### 9.1 主链路（评估→训练→分析）

```
assessment/ → assessment/（能力画像） → training-advice/ → training-record/ → dashboard/
```

说明：
- `assessment/` 产出原始分数和风险等级，同时生成/更新六领域能力画像
- `training-advice/` 基于能力画像确定训练目标和活动，生成训练方案
- `training-record/` 标准化记录每次训练数据，增量更新能力画像
- `dashboard/` 基于累积的训练记录和快照数据分析趋势，生成可视化看板

### 9.2 生活记录链路

```
life-record/ → dashboard/（生活模式分析）
```

说明：
- `life-record/` 积累到一定量后（建议7天以上），`dashboard/` 才能做有意义的模式分析
- 模式分析的发现会反馈到看板的 AI 洞察卡片中

### 9.3 行为分析链路

```
behavior-crisis/（ABC辅导）→ behavior-crisis/（ABC记录）→ behavior-crisis/（FBA）→ behavior-crisis/（BIP）→ dashboard/（干预效果追踪）
```

说明：
- ABC 辅导在用户首次做 ABC 记录或记录质量不高时自动触发，引导用户观察和描述行为的前因、具体表现和后果
- 需要积累至少5次以上同类行为记录才能做 FBA
- FBA 识别行为功能后，才能制定针对性的干预计划（BIP）
- 干预实施后持续记录 ABC 数据，`dashboard/` 对比干预前后变化评估效果
- behavior_records 与 life_records 分别建表，ABC 记录独立存储，便于跨角色读写和 FBA 分析

### 9.4 协作链路

```
任何业务函数的分析/报告 → collaboration/ → 接收方的 chat-casual/（解读）
```

说明：
- 医生生成的评估报告，通过 `collaboration/` 标准化格式并发送通知
- `collaboration/` 负责将专业内容转化为接收方能理解的语言
- 接收方打开通知后，`chat-casual/` 加载对应角色 prompt 解读内容

### 9.5 无依赖的独立能力

以下能力可以独立执行，不依赖其他函数的输出：
- `chat-casual/` 的家长居家指导（基于 children_memory + child_snapshots 直接回答）
- `behavior-crisis/` 的行为危机应对（紧急场景，直接给出指导）
- `training-advice/` 的课堂活动设计（可以独立设计，但有 IEP 时效果更好）
- `chat-casual/` 的能力发现导航（用户不清楚可用能力时直接触发）
