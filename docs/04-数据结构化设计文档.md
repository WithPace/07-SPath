## 1. 背景与目标

### 问题

v1.x 采用 FastAPI + Skill Script + MCP 三层架构：Agent 通过 MCP Server 调用 Python Skill Script，Skill Script 通过 supabase-py SDK 读写数据库。

存在的问题：
- MCP Server 作为中间层增加了部署复杂度和调试成本
- Python Skill Script 需要独立维护运行时环境
- FastAPI 后端与 Supabase 之间存在冗余的网络跳转
- intervention_frameworks 和 org_knowledge_base 表的结构化数据利用率低，不如直接融入 prompt 灵活
- 缺少对话消息持久化，无法回溯历史对话

### 目标

1. 将 Skill Script + MCP 架构迁移到 Supabase Edge Functions（TypeScript/Deno），减少中间层
2. 干预框架知识和机构知识库内容融入 Edge Function prompt，删除对应数据库表
3. 新增 chat_messages 表，持久化对话历史
4. 保留 Supabase（PostgreSQL）作为业务真源 + child_snapshots 双层快照架构
5. 保留 care_teams + RLS 行级权限体系
6. 保留快照刷新分级调度机制
7. 技术栈统一为 TypeScript：数据模型用 TypeScript interfaces / Zod 替代 Pydantic

## 2. 技术选型

| 组件 | 选择 | 理由 |
|------|------|------|
| 数据库 | Supabase（PostgreSQL） | 自带 REST API + 实时订阅 + RLS + Auth |
| 业务逻辑层 | Supabase Edge Functions（Deno） | 零冷启动、内置 supabase-js、与数据库同区域低延迟 |
| 运行时语言 | TypeScript (Deno) | Edge Function 原生支持，类型安全 |
| 数据库 SDK | supabase-js | Edge Function 内置，无需额外依赖 |
| 数据模型 | TypeScript interfaces / Zod | 类型安全 + 运行时校验 |
| 快照存储 | child_snapshots 表（JSONB） | 双层快照，Agent 高频语义读取 |

#### `child_snapshots` — Agent 快速读取缓存（双层快照）

| 字段 | 类型 | 说明 |
|------|------|------|
| child_id | UUID (FK → children) | 复合主键之一 |
| snapshot_type | VARCHAR(20) | 复合主键之一，取值 `short_term` 或 `long_term` |
| snapshot | JSONB | 快照内容（结构因 snapshot_type 而异，见下方） |
| refreshed_at | TIMESTAMPTZ | 最后刷新时间 |
| is_stale | BOOLEAN | 是否需要刷新，默认 false |

> 复合主键：`PRIMARY KEY (child_id, snapshot_type)`，每个孩子固定 2 条记录。

**短期快照（`short_term`）— 30 天滑动窗口，高频刷新**

包含字段：recent_sessions(30d)、recent_behaviors(30d)、recent_life(30d)、today_tasks、active_training_plan、unread_notifications、care_team

典型大小：20-80KB。刷新频率高（训练/生活/行为记录写入后触发）。

**长期快照（`long_term`）— 完整发展档案，低频刷新**

包含字段：basic、medical、memory、profile_current、profile_history(最近3个版本)、assessment_comparison(最近3次评估对比)、overall_trend(六领域长期趋势)、milestones

典型大小：50-150KB。刷新频率低（评估完成、画像更新、记忆变更时触发）。

**刷新策略：**

| 写入事件 | 触发刷新的快照层 | 说明 |
|---------|----------------|------|
| training_sessions INSERT | short_term | 训练记录只影响近期数据 |
| life_records INSERT/UPDATE | short_term | 生活记录只影响近期数据 |
| behavior_records（常规） | short_term | 常规行为记录只影响近期数据 |
| behavior_records（severity=urgent） | short_term + long_term | 危机事件同时影响两层 |
| assessments INSERT/UPDATE | short_term + long_term | 评估结果影响全局画像 |
| children_profiles INSERT | long_term | 画像更新影响长期档案 |
| children_memory UPDATE | long_term | 记忆变更影响长期档案 |
| care_teams 变更 | short_term + long_term | 权限变更影响两层 |

Agent 每次对话开始时调用 `getFullContext(childId)` 一次查询获取两层快照合并的完整上下文。按需也可单独调用 `getSnapshot(childId, 'short_term')` 或 `getSnapshot(childId, 'long_term')`。

### 公共模块目录

```
supabase/functions/_shared/
├── auth.ts              # JWT 验证 + care_teams 权限检查
├── snapshot.ts          # 双层快照读取（getFullContext / getSnapshot）
├── model-router.ts      # 大模型调用封装（Doubao/Kimi 切换 + 失败降级）
└── finalize.ts          # 收尾写入（operation_logs + snapshot_refresh_events 同事务）
```

#### `auth.ts` — 鉴权与权限检查

从请求 Header 提取 JWT，验证用户身份，查询 `care_teams` 表确认该用户对目标 child 的角色和权限。所有写入类 Edge Function 必须先调用 `auth.ts` 完成权限校验。

#### `snapshot.ts` — 双层快照读取

提供两个核心方法：
- `getFullContext(childId)` — 一次查询获取 short_term + long_term 两层快照，合并返回完整上下文
- `getSnapshot(childId, snapshotType)` — 读取指定层快照

快照读取失败或 `is_stale=true` 时，回源数据库查询并触发补刷。

#### `model-router.ts` — 大模型调用封装

统一封装 Doubao / Kimi 等大模型的调用接口，支持：
- 模型优先级配置与自动切换
- 调用失败时降级到备选模型
- 流式输出（Streamable HTTP）支持
- Token 用量追踪

#### `finalize.ts` — 收尾写入

所有写入操作完成后必须调用，在同一事务中完成：
- 写入 `snapshot_refresh_events`（pending 状态）
- 写入 `operation_logs`（db_write_status + outbox_write_status）

强约束：主业务写入与收尾写入必须同事务提交（Transactional Outbox），禁止跳过。

### 调用流程

```
前端 POST → orchestrator Edge Function（意图识别）
  → 分发到业务 Edge Function
    → auth.ts 鉴权 + 权限检查
    → snapshot.ts 读取双层快照
    → model-router.ts 调用大模型
    → 写入业务表
    → finalize.ts 收尾写入
    → Streamable HTTP 流式返回
```

强约束：
- 主业务写入与 `snapshot_refresh_events` 写入必须同事务提交（Transactional Outbox）。
- 任一环节失败必须写 `operation_logs`，保证可审计与可补偿。

### 周报与分析报告数据查询

**一周训练周报查询**：

```sql
-- 查询指定周的训练记录，按日分组
SELECT
  session_date as training_date,
  array_agg(DISTINCT target_skill) as skills,
  count(*) as session_count,
  avg(success_rate) as avg_success_rate,
  sum(duration_minutes) as total_minutes
FROM training_sessions
WHERE child_id = :child_id
  AND session_date >= :week_start
  AND session_date < :week_end
GROUP BY session_date
ORDER BY training_date;

-- 查询技能进度（六大领域）—— 从 children_profiles 最新版本读取
SELECT domain_levels
FROM children_profiles
WHERE child_id = :child_id
ORDER BY version DESC LIMIT 1;
```

**综合分析报告查询**：

```sql
-- 六大领域能力评分（从 domain_levels JSONB 读取）
SELECT domain_levels
FROM children_profiles
WHERE child_id = :child_id
ORDER BY version DESC LIMIT 1;

-- ABC行为模式统计
SELECT
  behavior_type,
  count(*) as frequency,
  mode() WITHIN GROUP (ORDER BY antecedent) as common_antecedent,
  mode() WITHIN GROUP (ORDER BY consequence) as common_consequence,
  behavior_function
FROM behavior_records
WHERE child_id = :child_id
  AND occurred_at >= :period_start
GROUP BY behavior_type, behavior_function
ORDER BY frequency DESC;

-- 季度趋势对比（基于训练记录的成功率）
SELECT
  date_trunc('quarter', session_date) as quarter,
  avg(success_rate) as avg_success_rate,
  count(*) as session_count
FROM training_sessions
WHERE child_id = :child_id
  AND session_date >= :start_date
GROUP BY quarter
ORDER BY quarter;
```

### 快照刷新事件映射

优先级评分模型：
`priority_score = risk*5 + visibility*3 + cross_role_dependency*2 - write_frequency_penalty*1`

| 事件源 | 条件 | 等级 | 触发快照层 | 队列 | 合并窗口 | 执行策略 | SLA |
|--------|------|------|-----------|------|---------|---------|-----|
| `care_teams` | 权限新增/变更/移除 | S1 | short_term + long_term | `snapshot_high` | 无 | 立即刷新，禁止合并 | P95 < 15秒 |
| `assessments` | insert/update | S1 | short_term + long_term | `snapshot_high` | 10秒 | 立即刷新 + 二次校验 | P95 < 15秒 |
| `reports` + `notifications` | type in ('suggestion','intervention','urgent_alert') | S1 | short_term | `snapshot_high` | 10秒 | 立即刷新 + 推送后复核 | P95 < 15秒 |
| `behavior_records` | `severity_flag='urgent'` | S1 | short_term + long_term | `snapshot_high` | 无 | 即时刷新，禁止合并 | P95 < 15秒 |
| `training_sessions` | insert | S2 | short_term | `snapshot_normal` | 60秒 | 合并窗口内批次刷新 | P95 < 5分钟 |
| `life_records` | insert/update | S2 | short_term | `snapshot_normal` | 120秒 | 合并窗口内批次刷新 | P95 < 5分钟 |
| `behavior_records` | 常规记录 | S2 | short_term | `snapshot_normal` | 60秒 | 合并窗口内批次刷新 | P95 < 5分钟 |
| `children_memory` | 常规字段更新 | S2 | long_term | `snapshot_normal` | 120秒 | 合并窗口内批次刷新 | P95 < 5分钟 |
| `children_profiles` | insert（新版本） | S2 | long_term | `snapshot_normal` | 60秒 | 合并窗口内批次刷新 | P95 < 5分钟 |
| `reports` | type in ('weekly','monthly','org_monthly') | S3 | long_term | `snapshot_batch` | 批次级 | 定时任务刷新 | 分钟级到小时级 |
| 批量回算任务 | backfill/reindex | S3 | short_term + long_term | `snapshot_batch` | 批次级 | 离峰全量重建 | 小时级 |

实现约束：
- snapshot 只读，不允许业务链路直接写入快照内容。
- 所有业务写入先落库，再由事件队列触发对应层快照刷新。
- 事件来源统一为 Database Trigger 或应用层 Hook，两者都写入同一快照事件队列。
- 快照读取失败或过期时，Agent 回源数据库并触发补刷。
- 双层快照独立刷新，互不阻塞；`is_stale` 标记用于标识待刷新状态。

### 监控指标（Observability）

| 指标名 | 说明 | 维度 | 目标/阈值 | 告警条件 |
|--------|------|------|----------|---------|
| snapshot_refresh_latency_ms | 刷新耗时（入队到成功） | snapshot_type, priority_level | S1: P95<15秒; S2: P95<5分钟 | 连续5分钟超阈值 |
| snapshot_queue_depth | 队列积压长度 | queue_name | high<100, normal<1000 | 超阈值持续10分钟 |
| snapshot_refresh_success_rate | 刷新成功率 | snapshot_type, priority_level | >=99.5% | 5分钟窗口<98% |
| snapshot_retry_count | 单事件重试次数 | event_id, queue_name | <=3 | >3进入死信并告警 |
| snapshot_stale_ratio | is_stale=true占比 | snapshot_type | <1% | >=3%持续15分钟 |
| snapshot_fallback_db_ratio | Agent回源数据库比例 | role, scene_type | <5% | >15%并持续上升 |
| snapshot_checksum_mismatch_count | 快照校验不一致次数 | snapshot_type, event_source | 0为目标 | 任意非0触发告警 |
| final_writeback_failure_rate | 收尾写入失败率（outbox或operation_logs失败） | action_name | <0.1% | >=0.5%持续5分钟 |

## 5. 权限体系（Supabase RLS）

### 角色体系

用户可拥有多角色（users.roles JSONB 数组）：
- `parent` — 家长
- `doctor` — 医生
- `teacher` — 特教老师
- `org_admin` — 机构管理员

### 权限链路

```
parent ──→ care_teams(role=parent) ──→ child 数据（完全控制）
doctor ──→ care_teams(role=doctor) ──→ child 数据（按 permissions）
teacher ──→ care_teams(role=teacher) ──→ child 数据（按 permissions）
org_admin ──→ org_members ──→ teachers ──→ care_teams ──→ child 数据（只读）
```

关键点：org_admin 不出现在 care_teams 中，通过机构→老师→照护团队的链路间接访问数据。

### 各表 RLS 策略

| 表 | parent | doctor | teacher | org_admin |
|----|--------|--------|---------|-----------|
| children | rw (owner) | r (care_team) | r (care_team) | r (via org chain) |
| children_medical | rw | rw (care_team) | 默认不可见，需授权 | r (via org chain) |
| children_memory | rw | r (care_team) | r (care_team) | r (via org chain) |
| children_profiles | rw | rw (care_team) | r (care_team) | r (via org chain) |
| assessments | r | rw (care_team) | r (按授权) | r (via org chain) |

### 家长自定义权限

家长作为 owner，可修改 care_teams 中医生/老师的 permissions 字段，精细控制每个域的读写权限。由数据库 RLS 强制执行，不依赖应用层检查。

### 按孩子独立控制数据访问

所有数据访问权限以孩子为单位独立控制（对应设置页的"数据访问权限"子页面）：

- 家长可查看自己孩子的所有训练记录（training_sessions）和行为记录（behavior_records），RLS 通过 `children.created_by = auth.uid()` 或 `care_teams(role='parent')` 匹配
- 医生/老师根据 `care_teams.permissions` 中各域的配置（r/rw/none）访问对应数据，RLS 在每次查询时实时校验权限字段
- 同一用户若照护多个孩子，每个孩子的权限配置互相独立，不会交叉影响
- 家长可随时在设置页调整某位医生/老师对某个孩子各数据域的访问级别，变更即时生效（care_teams UPDATE 触发 S1 快照刷新）
