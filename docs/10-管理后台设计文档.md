> 版本：v1.1 | 日期：2026-02-21 | 作者：StarPath Team

## 1. 概述

### 1.1 定位

星途AI运营管理后台是面向公司内部运营团队的 Web 端管理工具，**不面向终端用户（家长/医生/老师/机构）**。目标是让非技术背景的运营人员能够高效完成日常运营、数据查看、用户管理、内容维护等工作。

### 1.2 技术栈

| 层级 | 选型 | 说明 |
|------|------|------|
| 前端框架 | React 18 + TypeScript | 主流生态，组件丰富 |
| UI 组件库 | Ant Design Pro v6 | 开箱即用的中后台方案，内置图表、表格、表单 |
| 图表库 | @ant-design/charts（基于 G2） | 与 Ant Design 风格统一 |
| 数据层 | Supabase JS SDK | 直连同一个 Supabase 项目，复用现有数据库和 RLS |
| 认证 | Supabase Auth + 管理员专用角色 | 邮箱密码登录，与终端用户认证隔离 |
| 部署 | Vercel / Cloudflare Pages | 静态部署，通过 Supabase SDK 访问后端 |

### 1.3 管理员角色（RBAC）

| 角色 | 权限范围 | 典型使用者 |
|------|---------|-----------|
| 超级管理员 | 全部功能 + 系统设置 + 管理员账号管理 | CTO、技术负责人 |
| 运营人员 | 用户管理、内容管理、订阅管理、运营工具、机构管理 | 运营经理、运营专员 |
| 客服人员 | 用户查看（只读）、反馈管理、公告管理 | 客服专员 |
| 数据分析师 | 仪表盘、数据分析（全部只读） | 产品经理、数据分析师 |

> 管理员账号存储在 `admin_users` 表（新增），与终端用户 `users` 表完全隔离。

### 1.4 与主系统的关系

```
┌─────────────────────────────────────────────┐
│              Supabase 项目（共享）              │
│  ┌──────────┐  ┌──────────┐  ┌───────────┐  │
│  │PostgreSQL│  │Edge Func │  │  Storage   │  │
│  │ 20+张表  │  │ 12个函数 │  │  文件存储  │  │
│  └────┬─────┘  └────┬─────┘  └─────┬─────┘  │
│       │             │              │         │
├───────┼─────────────┼──────────────┼─────────┤
│       │             │              │         │
│  ┌────▼─────┐  ┌────▼─────┐       │         │
│  │ 终端App  │  │ 管理后台 │       │         │
│  │(Expo RN) │  │(React Web│       │         │
│  │ 用户侧   │  │ 运营侧)  │       │         │
│  └──────────┘  └──────────┘       │         │
└─────────────────────────────────────────────┘
```

管理后台通过 Supabase JS SDK 直连同一个数据库，使用独立的 RLS 策略（基于 `admin_users` 表的角色判断），确保管理员只能执行授权操作。

### 5.3 AI Prompt 管理说明

Prompt 是星途AI的核心资产，管理后台提供可视化编辑能力，但需严格管控：

- 编辑权限仅限超级管理员和运营人员
- 每次修改必须填写变更说明
- 修改后不立即生效，需「发布」操作确认
- 支持「预览」功能：用测试对话验证新 prompt 效果
- A/B 测试需设置实验周期和样本量，自动收集对比数据

### 6.2 模型成本估算

| 模型 | 输入价格 | 输出价格 | 日均调用量（预估） |
|------|---------|---------|------------------|
| Doubao-32k-pro | ¥0.008/千tokens | ¥0.008/千tokens | ~5000次 |
| Kimi 2.5 | ¥0.02/千tokens | ¥0.06/千tokens | ~2000次 |

管理后台自动按 token 用量计算每日/每月 AI 成本，超预算时触发告警。

### 6.3 对话质量抽检流程

1. 系统每日随机抽取 20 条对话（按 Edge Function 均匀分布）
2. 运营人员在后台查看对话上下文
3. 按四个维度打分（1-5分）：准确性、专业性、安全性、共情度
4. 低于 3 分的对话自动标记为「需优化」，关联到 Prompt 管理模块
5. 每周生成质量报告，追踪评分趋势

### 7.2 留存分析说明

留存是衡量产品健康度的核心指标，管理后台提供多维度留存视图：

| 留存类型 | 定义 | 目标值 |
|---------|------|--------|
| D1 次日留存 | 注册次日仍有操作的用户比例 | ≥ 40% |
| D7 七日留存 | 注册后第7天仍有操作的用户比例 | ≥ 25% |
| D30 月留存 | 注册后第30天仍有操作的用户比例 | ≥ 15% |
| 周活跃留存 | 本周活跃用户中，下周仍活跃的比例 | ≥ 60% |

留存曲线支持按角色、VIP等级、注册渠道分组对比，帮助定位不同用户群的留存差异。

### 7.3 转化漏斗详细定义

```
注册（手机号验证通过）
  │
  ▼  转化率目标：≥ 80%
创建孩子（完成孩子基本信息填写）
  │
  ▼  转化率目标：≥ 70%
首次对话（发送第一条消息并收到AI回复）
  │
  ▼  转化率目标：≥ 50%
首次评估（完成 M-CHAT 或其他评估）
  │
  ▼  转化率目标：≥ 10%
付费转化（升级为任意付费等级）
```

- 管理员通过邮箱+密码登录，与终端用户（手机号登录）完全隔离

| `quality_reviews` | 对话质量评审 | `message_id`, `reviewer_id`, `accuracy`, `professionalism`, `safety`, `empathy` | Phase 2 |
| `org_applications` | 机构入驻申请 | `org_name`, `contact`, `license_url`, `status`, `reviewer_id` | Phase 2 |

> 所有新增表均包含 `created_at` 和 `updated_at` 时间戳字段，并配置独立的 RLS 策略（仅 `admin_users` 角色可访问）。

### 13.2 AI 能力详细设计

#### ① 对话质量自动评分

**现状痛点**：人工抽检每日仅 20 条，覆盖率不足 0.2%，低质量对话容易漏检。

**AI 方案**：

| 维度 | 说明 |
|------|------|
| 实现方式 | 新增 Edge Function `admin-quality-review/` |
| 输入 | `chat_messages`（对话上下文）+ 对应角色 prompt + `child_snapshots`（孩子背景） |
| 输出 | 四维度评分（准确性/专业性/安全性/共情度，各 1-5 分）+ 改进建议文本 |
| 模型 | Kimi 2.5（需要严谨的多维度评判） |
| 触发 | pg_cron 每日凌晨批处理，评估前一天全部对话 |
| 写入表 | `quality_reviews`（自动评分标记 `reviewer_type = 'ai'`，人工复核标记 `reviewer_type = 'human'`） |

**工作流**：

```
pg_cron 每日 02:00 触发
  ↓
查询昨日全部 chat_messages（按 Edge Function 均匀采样，日量大时上限 200 条）
  ↓
逐条调用 Kimi 2.5 评分（批量并发，控制 QPS）
  ↓
写入 quality_reviews 表（reviewer_type = 'ai'）
  ↓
总分 < 3.0 的对话自动标记「需人工复核」
  ↓
运营人员在后台「对话质量」页面只需复核 AI 标记的低分对话
  ↓
每周自动生成质量趋势报告
```

**评分 Prompt 核心要求**：
- 准确性：回复内容是否符合循证依据，有无事实错误
- 专业性：术语使用是否恰当，建议是否具有可操作性
- 安全性：是否遵守安全边界（不做诊断、不推荐用药、高风险建议就医）
- 共情度：是否先回应情绪再给建议，语气是否符合角色定位

#### ② 运营日报 AI 摘要

**现状��点**：运营人员需要逐个查看仪表盘图表，手动总结当日情况。

**AI 方案**：

| 维度 | 说明 |
|------|------|
| 实现方式 | 新增 Edge Function `admin-daily-digest/` |
| 输入 | 聚合查询：`users`（新增/活跃）、`chat_messages`（对话量）、`training_sessions`（训练量）、`operation_logs`（AI 调用/错误/降级）、`feedbacks`（新增反馈）、付费数据 |
| 输出 | 结构化 JSON（关键指标 + 环比变化 + AI 文字摘要 + 需关注事项列表） |
| 模型 | Doubao-32k-pro（摘要生成任务，成本优先） |
| 触发 | pg_cron 每日 08:00 |
| 展示 | 管理后台仪表盘顶部「AI 运营简报」卡片 + 可选推送到企业微信/飞书 |

**输出示例**：

#### ③ 用户流失预警

**现状痛点**：仅靠"7 天未活跃标红"的简单规则，发现时用户往往已经流失。

**AI 方案**：

| 维度 | 说明 |
|------|------|
| 实现方式 | 新增 Edge Function `admin-churn-predict/` |
| 输入 | 近 30 天用户行为时序数据：登录频率、对话次数变化、训练记录频率、VIP 到期时间、最近一次评估距今天数 |
| 输出 | 高流失风险用户列表（风险分 0-100）+ 每个用户的流失原因标签 + 建议召回策略 |
| 模型 | Doubao-32k-pro（模式识别+文本生成） |
| 触发 | pg_cron 每日 06:00 批处理 |
| 展示 | 用户管理页面新增「流失预警」Tab，按风险分降序排列 |

**风险信号权重**：

| 信号 | 权重 | 说明 |
|------|------|------|
| 对话频率下降 > 50% | 高 | 近 7 天 vs 前 7 天对话次数对比 |
| 训练记录中断 ≥ 5 天 | 高 | 曾连续记录但突然中断 |
| 登录间隔拉长 > 3 天 | 中 | 此前日均登录的用户 |
| VIP 到期前 7 天未续费 | 中 | 付费用户续费预警 |
| 最近评估距今 > 90 天 | 低 | 可能已脱离康复周期 |
| 从未完成首次评估 | 低 | 激活不充分，价值感知弱 |

**建议召回策略**（AI 根据流失原因自动匹配）：

| 流失原因 | 召回策略 |
|---------|---------|
| 训练坚持困难 | 推送"简化版 5 分钟训练"建议 + 鼓励文案 |
| 付费到期犹豫 | 推送限时续费优惠 + 孩子进步数据回顾 |
| 功能使用单一 | 推送未体验功能引导（如看板、周报） |
| 新用户激活不足 | 推送新手引导 + 免费评估邀请 |

#### ④ 用户反馈智能分类

**现状痛点**：反馈标签靠人工打（功能建议/Bug/投诉/咨询/其他），逐条处理效率低。

**AI 方案**：

| 维度 | 说明 |
|------|------|
| 实现方式 | 嵌入反馈写入流程，反馈提交时实时调用 |
| 输入 | `feedbacks.content`（用户反馈原文） |
| 输出 | 自动分类标签 + 情感倾向（正面/中性/负面）+ 关键词提取 + 优先级评分（1-5） |
| 模型 | Doubao-32k-pro（分类任务，低延迟） |
| 写入 | 回写 `feedbacks.tags`（AI 分类）、`feedbacks.priority`（AI 优先级）、`feedbacks.ai_summary`（一句话摘要） |

**每周反馈摘要**（pg_cron 每周一生成）：

```
本周共收到 47 条反馈：
• Top 1：语音输入识别不准（12 条，负面情绪为主）→ 建议排查 ASR 服务
• Top 2：训练方案难度偏高（8 条）→ 建议 training-advice/ 增加难度分级
• Top 3：希望增加英文界面（5 条）→ 需求池记录，暂不处理
• 投诉类：2 条（AI 回复不专业），已关联 Prompt 优化
• 整体情感分布：正面 40% / 中性 35% / 负面 25%
```

#### ⑤ Prompt 优化建议

**现状痛点**：Prompt 编辑和 A/B 测试已有，但优化方向靠人工经验判断。

**AI 方案**：

| 维度 | 说明 |
|------|------|
| 实现方式 | 新增 Edge Function `admin-prompt-advisor/` |
| 输入 | `quality_reviews` 低分记录（近 30 天）+ 对应 `admin_prompts` 当前版本 + 低分对话的 `chat_messages` 上下文 |
| 输出 | 问题模式分析 + 具体 Prompt 修改建议（标注修改位置和理由） |
| 模型 | Kimi 2.5（需要长上下文分析和精准建议） |
| 触发 | 运营人员在 Prompt 管理页面手动点击「AI 优化建议」 |

**输出示例**：

```
分析了近 30 天 family-agent（家长端）的 23 条低分对话，发现以下模式：

1. 情绪崩溃场景共情不足（8 条，共情度平均 2.1 分）
   → 建议在 parent.md 第 3 段"回复结构"中增加：
     "当用户描述孩子情绪崩溃/行为失控时，必须先用 2-3 句话回应家长的情绪（疲惫、无助、焦虑），
      再给出应对建议。禁止直接跳到'您可以试试...'。"

2. 睡眠问题回复过于笼统（6 条，准确性平均 2.5 分）
   → 建议在 parent.md 增加睡眠相关的具体指导框架：
     "回复睡眠问题时，需要追问：入睡时间、夜醒次数、睡前仪式、感觉偏好，
      再给出个性化建议，避免泛泛而谈。"

3. 安全边界触发后的过渡生硬（5 条，共情度平均 2.3 分）
   → 建议修改安全边界提示语，从"我无法提供医疗诊断建议"改为
     "这个问题很重要，为了确保准确性，建议您带孩子去发育行为科面诊，
      我可以帮您准备就诊前需要整理的信息。"
```

#### ⑥ 智能告警归因

**现状痛点**：告警只展示"AI 超时率 > 5%"等阈值，运营不知道原因和处理方式。

**AI 方案**：

| 维度 | 说明 |
|------|------|
| 实现方式 | 告警触发时自动调用 AI 分析 |
| 输入 | 告警类型 + 近 1 小时 `operation_logs`（按函数/模型/错误类型聚合）+ `snapshot_refresh_logs`（快照相关告警） |
| 输出 | 根因分析 + 影响范围评估 + 建议处置方案 |
| 模型 | Kimi 2.5（需要数据分析和推理能力） |
| 展示 | 告警卡片展开后显示「AI 归因分析」区域 |

**输出示例**：

```
🔴 告警：AI 响应超时率 6.2%（阈值 5%）

AI 归因分析：
• 根因：assessment/ 函数在 14:00-15:00 调用量激增 180%（从 50 次/小时 → 140 次/小时），
  导致 Kimi 2.5 并发排队，P95 响应时间从 2.8s 升至 8.5s
• 触发原因：可能与今日公众号推文"免费 M-CHAT 筛查"引流有关
• 影响范围：约 45 个用户的评估对话出现明显延迟
• 建议处置：
  1. 短期：临时将 assessment/ 的 Kimi 降级阈值从 3s 调至 5s，减少不必要降级
  2. 中期：为 assessment/ 增加请求排队和限流机制
  3. 长期：评估是否需要为高峰时段预留 Kimi 2.5 专用额度
```

> 管理后台 AI 成本约占终端用户 AI 成本的 2-3%，边际成本极低但运营效率提升显著。
